# موسوعة القوانين اليمنية — تطبيق Flutter

تطبيق مجاني يعمل بدون إنترنت، يحتوي على **15 قانونًا يمنيًا** بصيغة نصية
منظمة (أبواب/فصول/مواد)، مع بحث شامل ونظام مفضلة، بواجهة عربية كاملة
باتجاه RTL.

تم بالفعل استخراج نصوص القوانين الـ15 من ملفات Word المرفقة وتحويلها إلى
قاعدة بيانات SQLite جاهزة (`assets/db/app_database.db`) تحتوي فعليًا على
**4,974 مادة قانونية** موزعة على القوانين الخمسة عشر — التطبيق جاهز للعمل
فورًا دون أي خطوة إضافية لإدخال البيانات.

## 2) تصميم قاعدة البيانات

القوانين اليمنية ليست كلها بنفس العمق الهرمي: بعضها (كالقانون المدني
وقانون الجرائم والعقوبات) يستخدم **كتاب ← قسم ← باب ← فصل ← مادة**،
وبعضها الآخر (كقانون العمل) يستخدم **باب ← فصل ← مادة** فقط، وبعضها
(كقانون التحكيم) يبدأ مباشرة بـ **فصل ← مادة**. لذلك صُمم جدول "الأبواب"
ليكون **شجريًا (self-referencing)** بحيث يستوعب "كتاب" و"قسم" و"باب" جميعًا
عبر `parent_bab_id`، وهذا هو الحل العملي الوحيد الذي يمثل كل القوانين
الـ15 بدون استثناءات ودون تكرار جداول لكل مستوى.

```sql
laws (
  id, name, short_name, order_num, articles_count
)

abwab (                         -- الأبواب (وتشمل أيضًا: كتاب / قسم عبر التداخل)
  id, law_id, parent_bab_id,    -- parent_bab_id NULL = مستوى أعلى (top-level)
  level,                        -- 'كتاب' | 'قسم' | 'باب'
  label, title, order_num
)

fusul (                          -- الفصول (تتبع بابًا دائمًا تقريبًا)
  id, law_id, bab_id, label, title, order_num
)

mawad (                          -- المواد (تتبع فصلاً، أو بابًا مباشرة إن لم يوجد فصل)
  id, law_id, fasl_id, bab_id, number, body, order_num
)

favorites ( id, mada_id, created_at )

mawad_fts   -- فهرس FTS5 لبحث نصي كامل سريع داخل نص المواد وأرقامها
```

**قاعدة الربط عند العرض:** كل مادة تنتمي إما إلى `fasl_id` (إن وجد فصل)،
أو مباشرة إلى `bab_id` (إن لم يوجد فصل)، أو لا شيء منهما (نادر جدًا: مادة
تسبق أول باب في القانون، مثل مادة التسمية الأولى في بعض القوانين).
شاشة `LawDetailScreen` تتعامل مع الحالات الثلاث تلقائيًا.

## 3) كيف تحوّلت ملفات Word إلى قاعدة بيانات

راجع `tools/build_db.py` — الخوارزمية باختصار:

1. تنظيف النص (إزالة حرف المطّ العربي "ـ" الذي كان يكسر التعرف على
   كلمات مثل "المــادة").
2. تقسيم النص إلى فقرات (فاصل: سطر فارغ).
3. تجاهل كل ما يسبق أول عنوان أو أول مادة (الديباجة الدستورية).
4. عند رصد فقرة عنوان (كتاب/قسم/باب/فصل) — عبر تعابير نمطية regex —
   يُغلق كل مستوى أعمق من المستوى الحالي في مكدس (stack)، ثم يُفتح عنصر
   جديد. عنوان الفصل الفرعي (مثل "أحكام عامة") يُستخرج إما من نفس
   السطر (بعد ":") أو من الفقرة التالية مباشرة إن لم تكن بداية مادة
   أو عنوان آخر.
5. عند رصد "مادة(رقم)" تُجمع كل الفقرات التالية (حتى أول عنوان/مادة
   جديدة) كنص المادة الكامل، مع الحفاظ على فقرات التعداد الفرعي
   (كـ"- الوزارة: ...").

### لتحديث/إعادة توليد قاعدة البيانات لاحقًا

```bash
cd tools
python3 build_db.py --src /path/to/folder_containing_the_15_docx_files \
                     --out ../assets/db/app_database.db
```

ثم في `lib/data/db_helper.dart` ارفع القيمة:

```dart
const int dbAssetVersion = 2;  // كانت 1
```

هذا يجعل التطبيق يستبدل تلقائيًا النسخة القديمة المخزّنة على جهاز
المستخدم عند أول تشغيل بعد التحديث (⚠️ ملاحظة: المفضلة مخزّنة في نفس
الملف، لذا تُصفَّر عند كل تحديث لقاعدة البيانات — إن أردت الحفاظ عليها
عبر التحديثات، انقل جدول `favorites` مستقبلاً إلى قاعدة بيانات منفصلة
صغيرة لا تُستبدل).

## 4) خطوات التشغيل (خطوة بخطوة)

**تحديث:** مجلد `android/` أصبح **مُضمَّنًا كاملاً** الآن في المشروع
(Gradle scaffolding + AndroidManifest + أيقونة التطبيق + شاشة بداية بلون
هوية التطبيق) — لم تعد بحاجة لتشغيل `flutter create` لإنشائه من الصفر.

### المتطلبات
- Flutter SDK (قناة stable) مُثبّت ومُهيّأ (`flutter doctor` بلا أخطاء حرجة).
- Android Studio أو VS Code مع إضافة Flutter.

### الخطوات

```bash
# 1) ادخل مجلد المشروع
cd yemen_laws_app

# 2) نزّل الحزم
flutter pub get

# 3) (مرة واحدة فقط) أكمل ملف gradle-wrapper.jar الثنائي المفقود.
#    هذا الملف الوحيد لم يُرفق لأنه ملف ثنائي (binary) لا يمكن كتابته
#    كنص. اختر إحدى الطريقتين التاليتين:
#
#    الطريقة أ (الأسهل): افتح المشروع في Android Studio → عند طلبه
#    "Gradle Sync" اضغط Sync، وسيقوم Android Studio تلقائيًا بإكمال
#    ملف الـ wrapper المفقود.
#
#    الطريقة ب (سطر الأوامر): إن كان لديك Gradle مثبتًا على جهازك:
cd android && gradle wrapper --gradle-version 8.6 && cd ..
#    أو ببساطة شغّل الأمر التالي مرة واحدة وسيُكمل فلاتر الملفات
#    القياسية الناقصة تلقائيًا دون المساس بأي من ملفاتك الحالية:
flutter create --platforms=android .

# 4) شغّل على المحاكي أو جهاز حقيقي
flutter run

# 5) لبناء ملف APK نهائي للتوزيع (موقّع مؤقتًا بمفتاح debug، يعمل للتجربة
#    والتوزيع المباشر خارج المتجر مباشرة بدون أي إعداد إضافي)
flutter build apk --release
# الناتج في: build/app/outputs/flutter-apk/app-release.apk
```

> **معرّف التطبيق (Application ID):** `com.yemenlaws.encyclopedia` — عدّله
> في `android/app/build.gradle` (سطري `namespace` و`applicationId`) وفي
> اسم حزمة `MainActivity.kt` قبل النشر إن رغبت بمعرّف مختلف.

### التوقيع للنشر الفعلي على متجر Google Play
نسخة `release` الحالية موقّعة تلقائيًا بمفتاح Debug (يعمل فورًا للتجربة
والتوزيع اليدوي APK، لكنه **غير صالح للرفع على المتجر**). لإنشاء مفتاح
توقيع حقيقي واستخدامه:

```bash
keytool -genkey -v -keystore ~/yemen-laws-key.jks -keyalg RSA -keysize 2048 \
        -validity 10000 -alias yemen_laws
```

ثم انسخ `android/key.properties.example` إلى `android/key.properties`
واملأ القيم الحقيقية:
```
storePassword=<كلمة المرور>
keyPassword=<كلمة المرور>
keyAlias=yemen_laws
storeFile=/المسار/الكامل/إلى/yemen-laws-key.jks
```

**لا حاجة لتعديل أي كود** — ملف `android/app/build.gradle` مُعدّ مسبقًا
ليكتشف وجود `key.properties` تلقائيًا ويستخدم مفتاحك الحقيقي بدلاً من
مفتاح Debug في أي بناء `release` لاحق. لا ترفع `key.properties` ولا ملف
`.jks` إلى أي مستودع عام (كلاهما مُستثنيان مسبقًا في `.gitignore`) — واحتفظ
بنسخة احتياطية آمنة منهما، لأن فقدان مفتاح التوقيع يمنعك من نشر أي تحديث
مستقبلي لنفس التطبيق على المتجر.



### تخصيص اسم التطبيق وأيقونته (تم إعداد الافتراضي مسبقًا)
- **اسم التطبيق الظاهر على الشاشة:** مضبوط بالفعل على "موسوعة القوانين
  اليمنية" في `android/app/src/main/AndroidManifest.xml` (`android:label`).
- **الأيقونة:** تم توليد أيقونة افتراضية (ميزان عدل ذهبي على خلفية كحلية)
  بكل الأحجام المطلوبة في `android/app/src/main/res/mipmap-*/`. لاستبدالها
  بشعارك الخاص: ضع صورة مربعة عالية الدقة (1024×1024 مثلاً) في
  `assets/icon/app_icon.png`، أضف حزمة `flutter_launcher_icons` في
  `pubspec.yaml`، ثم شغّل: `flutter pub run flutter_launcher_icons`.
- **خط عربي احترافي (Cairo/Tajawal):** حمّل الخط من Google Fonts، ضعه في
  `assets/fonts/`، أضف قسم `fonts:` في `pubspec.yaml` (متروك كمثال معلّق
  بداخله)، ثم فعّل `fontFamily: 'Tajawal'` في `lib/core/theme.dart`.

## 6) التوسّع المستقبلي

### أ) تحديث القوانين لاحقًا
- أعد تشغيل `tools/build_db.py` على النصوص المحدثة، وارفع `dbAssetVersion`
  كما في القسم (3) أعلاه. لا حاجة لتعديل أي كود آخر.
- إن رغبت بتحديث القوانين **عن بُعد دون إصدار تحديث جديد للتطبيق من
  المتجر**، يمكن مستقبلاً استبدال آلية "النسخ من assets" بتنزيل ملف
  `app_database.db` جديد من خادم (Firebase Storage أو أي رابط ثابت)
  عند توفر إنترنت، مع الإبقاء على النسخة المحلية كخيار احتياطي دون اتصال.

### ب) إضافة أحكام قضائية
أضف جدولاً جديدًا مستقلاً حتى لا يتأثر منطق القوانين الحالي:

```sql
CREATE TABLE ahkam_qadaiya (      -- الأحكام القضائية
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    title TEXT NOT NULL,          -- عنوان الحكم
    court_name TEXT,              -- المحكمة المُصدرة
    case_number TEXT,             -- رقم القضية
    ruling_date TEXT,             -- تاريخ الحكم
    related_madda_id INTEGER REFERENCES mawad(id),  -- ربط اختياري بمادة قانونية
    summary TEXT,                 -- ملخص الحكم
    full_text TEXT                -- النص الكامل
);
```

ثم أضف تبويبًا رابعًا في `main.dart` وشاشة `screens/rulings/` بنفس نمط
الشاشات الحالية (repository + provider اختياري + ListView من SectionTile).

### ج) أفكار إضافية جاهزة للتنفيذ بنفس البنية
- **الوضع الليلي (Dark Mode):** أضف `AppTheme.dark` في `theme.dart` وفعّله
  عبر `ThemeMode` مرتبط بإعداد يُحفظ محليًا (`shared_preferences`).
  - **حجم الخط القابل للتعديل:** أضف شريط تمرير في الإعدادات يضبط
  `textScaleFactor` عبر `MediaQuery`.
- **مقارنة نسخ القانون (قبل/بعد التعديل):** يمكن استغلال حقل `number`
  الذي يدعم صيغًا مثل `"45 مكرر"` لتمييز المواد المُستحدثة بالتعديلات.

## 7) ملاحظات فنية مهمة

- **الملفات المصدر ليست ملفات .docx حقيقية (ZIP)** بل ملفات نصية عادية
  UTF-8 محفوظة بامتداد `.docx`. السكربت يتعامل مع الحالتين تلقائيًا
  (يحاول `pandoc` أولاً إن كان الملف ZIP فعليًا، وإلا يقرأه كنص مباشرة).
- تكرار بعض أرقام المواد داخل نفس القانون (مثل "مادة (1)" تظهر أكثر من
  مرة) هو أمر **متوقع وصحيح قانونيًا**: يحدث عند وجود قوانين تعديل مُلحَقة
  بنهاية النص الأصلي (كل قانون تعديل يبدأ ترقيمه من (1) من جديد: "المادة
  (1): تُعدَّل المادة (16) من القانون رقم...").
- تأكد بعد التشغيل الأول من مطابقة أعداد المواد الظاهرة في الشاشة
  الرئيسية مع مصادرك الرسمية، وأبلغني بأي انحراف في قانون معيّن لأصلح
  الحالة الخاصة به في `build_db.py`.
